# CLAUDE.md — постоянные инструкции и обзор проекта OreVision

Этот файл Claude Code читает автоматически при каждом запуске в этой папке.
Здесь — краткий, но полный обзор текущего состояния кода и правила работы.

## Что это
**OreVision** — локальный веб-сайт (Streamlit) для анализа полированных шлифов
руды: по снимку/панораме определяется сорт руды и строится цветная карта
(тальк / обычные срастания / тонкие срастания), считаются метрики и rule-based
класс руды, есть экспорт и active learning (правки эксперта → патчи для дообучения).

Проект **самодостаточный, в одном репозитории**: модель классификации сортов
вшита внутрь. Внешних репозиториев/сервисов для работы не нужно.

> Историческая справка: раньше репозиторий вёлся двумя ИИ-агентами (потоки A/B,
> см. `AGENTS.md`) и работал в MOCK-режиме до готовности модели. **Сейчас это
> неактуально**: разделения на потоки нет, MOCK-режим удалён, модель вшита.

## Архитектура (текущее состояние)

Поток данных:
```
загрузка → ml_client.analyze → (метрики) → (классификация) → UI/экспорт/active learning
```

- **Встроенная модель** — `ml_service/`:
  - `grade_unfreeze_best.pth` (99.7 МБ, вшит в git) — se_resnext50_32x4d/MicroNet,
    macro-F1 = 0.944; классифицирует квадратные тайлы (train-FOV 512 px) в 3 сорта.
  - `bg_head_best.pth` (~10 КБ, вшит в git) — голова-детектор ФОНА `Linear(2048→1)`
    поверх ТОГО ЖЕ замороженного энкодера (пакет `../ore_4class_package`). Добавляет
    4-й класс «фон/нерудная матрица» (код контракта **0**). Детекция идёт **по
    тайлам** каскадом (`model.infer_batch_cascade`) на любых снимках, включая
    панорамы: `sigmoid(bg_head) > порога → код 0`, иначе argmax сорта. ⚠️ Вшитый
    чекпоинт обучался на снимках-целиком и на 512-px тайлах панорамы переоценивает
    фон — оставлен под ручное дообучение под тайлы (`ORE_ML_BG_CKPT`; пусто →
    режим «3 класса»).
  - `model.py` — загрузка весов (grade + bg_head) + батч-инференс (torch лениво).
  - `infer.py` — тайлинг панорамы → JSON строго по `API_CONTRACT.md` (contract v2:
    блочная маска + `patch_grid` + `objects`). Сопоставление классов модели с
    кодами контракта: **talc→3, ordinary→1, fine→2**, плюс тайл-фон → **0**.
  - `server.py` — опциональный Flask-сервис (`/health`, `/analyze`) для режима real.
  - `train.py` — **модуль активного обучения**: дообучает вшитую модель на
    ImageFolder-экспорте патчей (`export_active_learning_patch`) и выдаёт новый
    чекпоинт той же архитектуры (грузится через `ORE_ML_CKPT`). Весь цикл
    обучения теперь внутри репозитория; `reference/` — аудиторские копии
    оригинальных скриптов из `../ore_classification`.
- **Режимы ML** (`src/config.py:ML_MODE`, env `OREVISION_ML_MODE`):
  - `local` (по умолчанию) — модель грузится и считает **в процессе сайта**
    (`src/ml_client.py:_analyze_local` → `ml_service.infer`). Один `streamlit run`,
    без отдельного сервера. Нужен torch-стек (в `requirements.txt`).
  - `real` — HTTP на `http://localhost:8001/analyze` (поднять `ml_service/server.py`).
  - **MOCK-режима нет.** `mock_ml/generator.py` остался ТОЛЬКО как тест-фикстура.
- **Логика** — `src/`: `contract.py` (валидатор ответа), `metrics.py` (площади→%),
  `classification.py` (класс руды), `pipeline.py` (склейка), `quantizer.py`
  (область эксперта → патчи train-размера), `reports.py`/`gis_export.py`/`storage.py`/
  `dataset_storage.py` (экспорт и хранение).
- **Косметическая подсветка талька** — `talc_cosmetic/` (вендор `../deploy_segment`
  + `../talc_red_zones`, внесён в репозиторий ради самодостаточности): опциональный
  ВИЗУАЛЬНЫЙ слой поверх снимка, включается галочкой на главной ПОСЛЕ основного
  блочного анализа. Пайплайн (чистый CV, numpy+scipy): палитровая сегментация
  (mean-field Potts, `seglib.py`; палитра микроскопа `панорамная`/`жёлтая` —
  авто по ошибке квантования) → тальк = тёмный «чёрный» класс → density-сборка
  сплошной «области оталькования» (`talc_region.py`) → красная зона на оригинале.
  `overlay.compute_talc_overlay` считается на разрешении показа; на предсказание
  модели, метрики и класс руды **не влияет**.
- **UI** — `OreVision.py` (главный экран) + `pages/` + `ui/`:
  - `ui/annotator_frontend/` + `viewer.annotator()` — аннотатор в духе GUI
    matplotlib: прямоугольный **зум** (выделение растягивается на кадр),
    тумблер **оригинал/оверлей**, **лассо** прямо на приближении.
  - Порядок на главной после инференса: **Экспорт данных → Активное обучение**.
  - Страницы: `2_Разметка_эксперта` (ручной аннотатор, НЕ активное обучение),
    `3_История_образцов`, `1_Пакетная_обработка`, `4_Логи`.
  - **Активное обучение** — замкнутый цикл на самой OreVision (`src/active_learning.py`):
    эксперт правит разметку (лассо) → модель **дообучается сразу** (быстро — только
    «голова», энкодер заморожен, признаки кешируются; `ml_service/train.py:quick_finetune`)
    → снимок **переанализируется** дообучённой моделью (`infer.analyze_image(params={"ckpt": …})`)
    → на странице показывается «до/стало». Дообучение накопительное, версии весов —
    в `data/active_learning/` (в git не коммитятся). Страница «Разметка эксперта» к
    активному обучению отношения не имеет — это отдельный ручной аннотатор.

## Ключевые инварианты
- **Один контракт** (`API_CONTRACT.md`, v2): local и real отдают идентичный JSON,
  поэтому переключение режима ничего не ломает.
- **Сэмплинг патчей** (`src/quantizer.py`) робастен: кадр ≈ одного train-FOV,
  залитый одним классом → возвращается та же картинка; меньший — апскейл один раз;
  2-3 класса в маленьком кадре → мажоритарный класс; никогда не падает.
- **Память батча**: сырые байты картинок не держим в session_state — сбрасываем
  на диск, работаем по пути (иначе OOM на нескольких панорамах).
- **Размер фото не ограничен**: гигапиксельные панорамы поддержаны (показ через
  уменьшенное превью; `maxUploadSize=5 ГБ`).

## Как запускать
```bash
pip install -r requirements.txt     # включает torch-стек встроенной модели
streamlit run OreVision.py          # http://localhost:8501 (режим local)
pytest -q                           # 91 тест
```
Вынести инференс на отдельную (GPU) машину: `python ml_service/server.py` +
`OREVISION_ML_MODE=real`.

Дообучить модель на правках эксперта (active learning): в UI экспортировать
патчи (формат «Патчи для patch-модели»), затем
`python -m ml_service.train --patch-export <экспорт> --save-ckpt ml_service/grade_al_finetuned.pth`
и подключить новый чекпоинт через `ORE_ML_CKPT`.

## Правила работы с git
1. Не коммить прямо в `main` — только через ветку `stream-*/<задача>` и Pull Request.
2. Перед коммитом держать зелёными:
   - `python -m py_compile OreVision.py batch_process.py src/*.py ui/*.py mock_ml/*.py ml_service/*.py pages/*.py`
   - `pytest -q`
3. Данные из `data/` не коммить (в `.gitignore`). Веса `*.pth` тоже в `.gitignore`,
   КРОМЕ вшитой модели `ml_service/grade_unfreeze_best.pth` (явное исключение).
4. Спрашивать подтверждение перед разрушительными командами (force-push, reset --hard).
