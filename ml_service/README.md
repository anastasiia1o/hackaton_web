# orevision-ml — реальный ML-сервис (классификация сортов руды)

Это **сторона ML** из [`API_CONTRACT.md`](../API_CONTRACT.md): HTTP-сервис поверх
реальной модели `grade_unfreeze_best.pth` (backbone se_resnext50_32x4d, обучен на
MicroNet; macro-F1 = 0.944 на honest held-out). Модель приехала из соседнего
репозитория `../ore_classification` (Задача 3 «Скажи мне, кто твой шлиф»).

Сайт при этом **не меняется**: он уже умеет ходить в ML по HTTP
(`src/ml_client.py`, режим `real`). Переключение mock → real — одной настройкой.

## Что делает
Классифицирует квадратные тайлы панорамы (train-FOV 512 px) в один из 3 сортов и
отдаёт ответ **строго по контракту v2** (блочная маска + `patch_grid`):

| выход модели | сорт руды          | код класса контракта |
|-------------|--------------------|----------------------|
| 0 talc      | Оталькованная      | **3** (Тальк)        |
| 1 ordinary  | Рядовая            | **1** (Обычные срастания) |
| 2 fine      | Труднообогатимая   | **2** (Тонкие срастания)  |

Проценты и итоговый класс руды считает **сайт** (rule-based, `src/metrics.py` +
`src/classification.py`) — ровно как оговорено в контракте. Сервис отдаёт только
«сырые факты о пикселях».

### Класс «Фон» — голова-детектор `bg_head_best.pth`
Пакет `../ore_4class_package` добавил к сорту руды 4-й класс — **фон / нерудная
матрица** (код контракта **0**). Это отдельная голова `bg_head` — `Linear(2048→1)`
поверх ТОГО ЖЕ замороженного энкодера (`grade_unfreeze_best.pth` при этом
побайтово прежний, macro-F1 = 0.944); `bg_head` весит ~10 КБ и вшит в git.

Детекция фона идёт **по тайлам** каскадом за один проход энкодера
(`infer_batch_cascade`): из общего эмбеддинга `sigmoid(bg_head) > порога` →
тайл = код 0, иначе `argmax` головы сорта. Работает на **любых** снимках, включая
панорамы (по тайлам).

⚠️ Вшитый `bg_head_best.pth` обучался на снимках-целиком и на 512-px тайлах
панорамы **переоценивает фон** (размечает фоном большую часть даже заведомо
рудной панорамы). Поэтому голова фона **дообучается в активном обучении** вместе
с головой сорта — `train.py:quick_finetune_bg` (BCE на кешированных признаках
frozen-энкодера; правки эксперта «фон → руда» = метка 0, «→ фон» = метка 1,
якоря держат оба класса). Голова уверенная (~0.99), поэтому учится дольше
(≈150 эпох), но сдвигается: на панораме 10.jpg 12 правок «это руда» опускают
долю фона с 97.6 % до ~30 %. Дообученный чекпоинт подключается через
`ORE_ML_BG_CKPT` (или автоматически из `data/active_learning/…__al_bg_vN.pth`).
Полностью выключить фон (режим «3 класса»): пустой `ORE_ML_BG_CKPT`.

Переменные окружения: `ORE_ML_BG_CKPT` (путь к `bg_head`; пусто → отключить фон),
`ORE_ML_BG_THRESHOLD` (порог sigmoid, 0.5).

## Эндпоинты
- `GET  /health`  → `200 OK`, индикатор доступности в UI (модель не грузит).
- `POST /analyze` → `multipart/form-data` (`image`, опц. `params`-JSON) → JSON по
  `API_CONTRACT.md`.

## Запуск

```bash
# 1. Зависимости сервиса (ставятся ОТДЕЛЬНО от сайта; torch — тяжёлый).
pip install -r ml_service/requirements.txt
#   CPU-сборка torch:  pip install torch --index-url https://download.pytorch.org/whl/cpu

# 2. Веса модели должны лежать здесь: ml_service/grade_unfreeze_best.pth
#    (файл в git не хранится — *.pth в .gitignore; берётся из ../ore_classification).

# 3. Поднять сервис (по умолчанию :8001).
python ml_service/server.py

# 4. В ДРУГОМ терминале — сайт в real-режиме:
OREVISION_ML_MODE=real streamlit run OreVision.py
```

В сайдбаре сайта индикатор ML станет зелёным (он опрашивает `GET /health`).

## Настройки (переменные окружения)

| Переменная              | По умолчанию                    | Смысл                                    |
|-------------------------|---------------------------------|------------------------------------------|
| `ORE_ML_PORT`           | `8001`                          | порт сервиса (контракт ждёт 8001)        |
| `ORE_ML_CKPT`           | `ml_service/grade_unfreeze_best.pth` | путь к весам                        |
| `ORE_ML_TILE`           | `512`                           | сторона тайла (train-FOV модели)         |
| `ORE_ML_BATCH`          | `32`                            | тайлов в GPU-батче                       |
| `ORE_ML_MODE_INFER`     | `grid`                          | `grid` (быстро) или `slide` (качественнее, 50 % перекрытие) |
| `ORE_ML_CONF_THRESHOLD` | `0.0` (выкл)                    | тайлы с conf < τ помечать кодом 4 (неуверенно) |
| `ORE_ML_OUTPUT_DIR`     | `ml_service/outputs`            | куда пишутся PNG маски/сетки             |

Те же значения можно передать пер-запрос в `params` (например
`{"mode": "slide", "tile": 512}`).

## Скорость
grid ≈ 9 c, slide ≈ 33 c на панораму 10 000×10 000 (GPU). Требование жюри
(≤ 5 мин на 10 000×10 000) выполняется с запасом. На CPU — медленнее, но работает.

## Проверено на реальных данных
Живой прогон сервиса (torch 2.12 + smp 0.3.4, CUDA) на панораме `Панорамы/10.jpg`
(14 999×10 391): `POST /analyze` → HTTP 200 за ~21 c (grid), сетка 21×30, 74 объекта.
Распределение классов — тальк **24.6 %** / обычные **1.9 %** / тонкие **73.5 %** —
воспроизводит эталон ML-команды для этой панорамы (22.4 / 2.5 / 75.1 %), что
подтверждает корректность сопоставления классов и всей интеграции.

## Проверка без torch
Форму ответа (совпадение с контрактом) можно проверить, не устанавливая torch —
инференс модели заглушается:

```bash
python ml_service/test_contract_shape.py
```

Тест прогоняет ответ сервиса через валидатор сайта (`src/contract.py`) и весь
пайплайн (`metrics` → `classification`).

## Дообучение (active learning)

`train.py` замыкает цикл активного обучения **внутри этого репозитория** (раньше
обучение жило только в `../ore_classification`). Он ест ImageFolder-экспорт патчей
из редактора разметки (`src/dataset_storage.export_active_learning_patch` →
`imgs/<класс>/*.jpg` + `manifest.csv` с весами) и выдаёт новый чекпоинт **той же
архитектуры**, который без правок грузит `model.py`.

```bash
# 1. В UI (страница «Разметка эксперта») разметить области и экспортировать
#    формат «Патчи для patch-модели (ImageFolder + weights)».
# 2. Дообучить вшитую модель на этих патчах:
python -m ml_service.train \
    --patch-export data/datasets/<id>/exports/active_learning_patch/<export_id> \
    --save-ckpt ml_service/grade_al_finetuned.pth
# 3. Подключить новый чекпоинт (local или real режим):
ORE_ML_CKPT=ml_service/grade_al_finetuned.pth python ml_service/server.py
```

Стратегия — как в оригинальном `train_grade_classifier_v2.py` (старт с вшитого
чекпоинта, разморозка энкодера с `encoder_lr = lr·0.1`, cosine-расписание,
взвешенная CE по обратной частоте класса) плюс `WeightedRandomSampler` по весам
патчей. Можно подмешать «эталонный» датасет по сортам через `--base-dataset`
(ImageFolder: подпапки `talc`/`ordinary`/`fine`). Рядом с чекпоинтом пишется
`*.report.json` (метрики, состав трейна, история эпох). Коды классов размётки
(контракт 1/2/3) переводятся в индексы выхода модели (talc→0, ordinary→1, fine→2).

## Что в этой папке
- `server.py` — Flask-приложение (`/health`, `/analyze`).
- `model.py` — загрузка весов + батч-инференс тайлов (ленивый импорт torch).
- `infer.py` — тайлинг панорамы + сборка JSON по контракту.
- `train.py` — дообучение вшитой модели на патчах active learning (CLI).
- `test_contract_shape.py` — проверка контракта без torch.
- `reference/` — исходные скрипты и `ОПИСАНИЕ_КОДА.txt` от ML-команды (для аудита).
- `grade_unfreeze_best.pth` — веса (в git не коммитятся).
