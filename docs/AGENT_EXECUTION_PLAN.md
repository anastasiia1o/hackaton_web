# Агентский план выполнения OreVision

Исходный план был написан так, будто его руками выполняет команда людей.
На деле код пишет ИИ-агент в агентском формате. Ниже — перекладка плана под
эту реальность: что делает агент автоматически, что остаётся за человеком
(решения, доступы, показ жюри), и как два агента работают параллельно.

---

## 0. Что изменилось при переходе «человек → агент»

| Было (для людей)                              | Стало (агентский формат)                                  |
|-----------------------------------------------|-----------------------------------------------------------|
| «изучите, как делают сайты»                   | агент сразу генерирует рабочий код и объясняет его         |
| «настройте окружение по шагам»                | агент даёт готовые команды + `requirements.txt` + Docker  |
| «договоритесь в чате о контракте»             | контракт зафиксирован файлами (`API_CONTRACT.md`, SEAM)   |
| «менеджер раздаёт задачи»                     | задачи и статусы живут в `docs/coordination/BOARD.md`     |
| «созвон для синхронизации»                    | синхронизация через `HANDOFF.md` + git-ветки              |
| ручное тестирование                            | агент прогоняет `pytest` и сквозной smoke-тест перед PR    |

Роль человека сжимается до: принять архитектурные развилки, создать GitHub и
выдать доступы, подключить настоящий ML, записать видео-демо и показать жюри.

## 1. Выбор стека (анализ 4 вариантов) → выбран Streamlit + Python

Требование из задания — критически сравнить и выбрать один стек. Итог:

| Вариант                     | Плюсы                                              | Минусы для нас                                        |
|-----------------------------|----------------------------------------------------|-------------------------------------------------------|
| **1. Streamlit + Python** ✅ | один язык, UI за десятки строк, мгновенный локальный запуск, идеально для демо | не enterprise-грейд, тяжёлая кастомизация UI          |
| 2. Streamlit + FastAPI       | «правильное» разделение фронта и API               | вдвое больше кода и точек отказа — лишнее для 1.5 суток|
| 3. React + FastAPI           | максимально гибкий, промышленный                   | нужен опыт JS/сборки — команда без web-опыта не успеет |
| 4. Gradio + Python           | ещё быстрее для ML-демо                            | слабее layout/страницы/таблицы под наш сложный экран  |

**Выбор: вариант 1 — только Streamlit + Python.**
- *Почему лучше для нас:* один язык (Python), геолог открывает `localhost:8501`,
  весь сквозной сценарий собирается за часы, а не дни.
- *Компромиссы:* Streamlit не для сложных SPA-интерфейсов и высоких нагрузок;
  вьюер гигапиксельных панорам придётся усиливать сторонними компонентами.
- *Почему не другие сейчас:* FastAPI/React добавляют инфраструктуру, которая
  не приносит баллов на MVP; Gradio слабее по многостраничному интерфейсу.
- *Как развить после хакатона:* логика уже изолирована в `src/` и общается с
  ML по HTTP-контракту — фронт можно заменить на React, а `src/` обернуть в
  FastAPI, не переписывая ядро. То есть путь к варианту 3 открыт и дешёв.

> ML-сервис — отдельный процесс на FastAPI (`:8001`) в **чужом** репозитории.
> Мы к нему только клиент. Это не противоречит выбору «Streamlit-only» для сайта.

## 3. Как два агента идут параллельно (агентский workflow)

1. Оба агента стартуют с общего `main`, где уже лежит рабочий фундамент.
2. Агент читает `AGENTS.md` + свой `PLAN_AGENT_*.md` + свежий `HANDOFF.md`.
3. Берёт задачу из `BOARD.md`, создаёт ветку своего потока.
4. Работает в границах своих файлов (пересечение минимально по дизайну).
5. Проверки (`py_compile`, `pytest`, ручной запуск) → PR → мерж в `main`.
6. Обновляет `BOARD.md`/`HANDOFF.md`. Второй агент подхватывает свежий `main`.

Смысл: агенты почти не редактируют одни и те же файлы, а всё «межпоточное»
общение идёт через 4 markdown-файла и `schemas.py`. Это и есть замена чату.

## 4. Что человек делает руками (агент не может)

- Создать GitHub private-repo, выдать доступ второму участнику/агенту.
- Установить Docker Desktop (если нужен demo-режим через Compose).
- Подключить реальный ML-сервис (адрес/доступ к `orevision-ml`).
- Записать видео-демо (≤5 мин) и сделать презентацию для сдачи.
- Загрузить архив на облачный диск по требованиям формата сдачи.
