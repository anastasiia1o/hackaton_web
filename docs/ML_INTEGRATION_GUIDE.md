# Руководство по интеграции ML-сервиса с OreVision

Этот документ — для **команды ML** (`orevision-ml`) и её ИИ-агента.
Он объясняет простым языком, что именно ваш сервис должен принимать и
возвращать, чтобы сайт OreVision смог его использовать без единой правки.

Формальная короткая спецификация — в [`API_CONTRACT.md`](../API_CONTRACT.md).
Здесь — то же самое, но с объяснениями, примерами и типичными ошибками.

---

## 1. Что от вас требуется в двух предложениях

Поднять HTTP-сервис на `http://localhost:8001` с одним рабочим методом
`POST /analyze`. Он принимает изображение шлифа и возвращает **JSON с «сырыми
фактами о пикселях»**: маску классов (картинкой), список найденных объектов и
карту уверенности. Проценты и класс руды считать **не нужно** — это делает сайт.

> Золотое правило разделения: **ML говорит «где какие пиксели», сайт говорит
> «что это за руда».** Не возвращайте `talc_percent`, `ore_class` и т.п. —
> сайт применяет геологические правила сам, прозрачным кодом.

---

## 2. Эндпоинты

### `POST /analyze` (обязательный)

- Тип запроса: `multipart/form-data`
- Поля:
  - `image` — файл изображения (TIFF / PNG / JPEG). **Обязательно.**
  - `params` — JSON-строка с параметрами инференса. Необязательно.
- Ответ: `application/json` (структура — раздел 4).

### `GET /health` (желательный)

Возвращает `200 OK` (можно `{"status":"ok"}`). Сайт по нему показывает
зелёный/красный индикатор доступности ML. Если метода нет — не критично.

---

## 3. Коды классов — ЗАФИКСИРОВАНЫ, менять нельзя

Маска — это изображение, где **значение каждого пикселя = код класса**:

| Код | Класс                     | Что это на шлифе                                              |
|----:|---------------------------|--------------------------------------------------------------|
| 0   | фон / нерудная матрица    | силикатная/оксидная основа                                    |
| 1   | обычные срастания         | крупные светлые сульфиды с **небольшим** замещением           |
| 2   | тонкие срастания          | сульфиды со **значительным** замещением серой/тёмной фазой    |
| 3   | тальк                     | тёмная рассеянная фаза в нерудных зонах                       |
| 4   | артефакт / исключено      | царапины, грязь, зоны вне анализа                             |

Важнейшее для качества: **различие классов 1 и 2 — это степень замещения, а не
размер включения.** Именно ваша модель решает, обычное это срастание или тонкое.
Сайт лишь суммирует площади — доклассификации по размеру он НЕ делает.

Маску сохраняйте как **8-битный PNG (grayscale, режим `L`)**. Значения пикселей
— ровно 0..4, без сглаживания/интерполяции (никаких промежуточных 127 и т.п.).

## 4. Формат ответа (единый JSON)

```jsonc
{
  "model_version": "unet-r34-0.3.1",      // строка версии вашей модели
  "inference_time_ms": 4213,              // сколько считали, целое число мс
  "inference_params": { "tile": 1024 },   // с какими параметрами (эхо params)
  "image_size": { "width": 10000, "height": 8000 },  // размер В ПИКСЕЛЯХ

  "mask": "/abs/path/pred_mask.png",      // БЛОЧНАЯ PNG-маска (v2): пиксель = код
                                          //   класса; = patch_grid.labels, апскейл nearest
  "confidence_map": "/abs/path/conf.png", // grayscale-PNG, ярче = увереннее (0..255)

  "patch_grid": {                          // v2: сырой квантованный вывод patch-модели
    "tile": 2272, "stride": 2272,          //   окно/шаг патча в пикселях исходника
    "rows": 6, "cols": 9, "origin": [0, 0],
    "labels": "/abs/path/grid_labels.png", //   PNG rows×cols, пиксель = код класса
    "conf":   "/abs/path/grid_conf.png"    //   PNG rows×cols, яркость = уверенность
  },

  "class_legend": {                        // расшифровка кодов (ровно как в §3)
    "0": "фон / нерудная матрица",
    "1": "обычные срастания",
    "2": "тонкие срастания",
    "3": "тальк",
    "4": "артефакт / исключено"
  },

  "objects": [                             // список найденных включений
    {
      "id": 0,                             // уникальный int
      "class": 1,                          // код класса 1..4 (0 обычно не выделяют)
      "bbox": [x, y, w, h],                // прямоугольник в пикселях, левый-верх + ш/в
      "area_px": 12045,                    // площадь объекта в пикселях, int
      "confidence": 0.94                   // уверенность 0..1
    }
  ],

  "warnings": [                            // необязательные текстовые предупреждения
    "неравномерное освещение в правом верхнем углу"
  ]
}
```

### Пояснения к полям

- `image_size` **должен совпадать** с реальными размерами PNG-маски, иначе сайт
  отвергнет ответ (валидатор это проверяет — см. §6).
- `mask` и `confidence_map` — это **пути к файлам**, а не массивы чисел в JSON.
  Так мы не раздуваем JSON гигапиксельными данными. Про пути — см. §5.
- `patch_grid` (v2, patch-classification) — сырой квантованный вывод: сетка
  патчей `rows×cols`. `mask`/`confidence_map` — это ровно `labels`/`conf`,
  растянутые nearest'ом до `image_size` (поэтому `mask` теперь БЛОЧНАЯ). Поле
  опционально: без него ответ валиден (только предупреждение), но для patch-
  модели отдавайте его. Валидатор сверяет `size(labels)==rows×cols` и
  `mask==nearest-upsample(labels)`. См. `docs/PATCH_AL_REDESIGN.md`.
- `objects` может быть пустым списком `[]`, если ничего не найдено. Не `null`.
- `confidence` в объектах и яркость в `confidence_map` — ваша оценка уверенности.
  Сайт использует их, чтобы честно уводить спорные случаи в «экспертную проверку».
- Лишние поля в JSON не запрещены — сайт их проигнорирует. Но обязательные
  поля выше должны присутствовать всегда.

## 5. Где лежат файлы маски и confidence (важно для запуска)

Сайт должен **прочитать** PNG-файлы, пути к которым вы вернули. Есть два случая:

- **Сайт и ML на одной машине / общий диск (наш MVP).** Возвращайте абсолютный
  путь к файлу на общей файловой системе, например
  `C:/orevision/shared/pred_mask.png` или `/data/shared/pred_mask.png`.
  Договоритесь о единой общей папке.
- **ML в отдельном контейнере без общего диска.** Тогда одного пути мало — сайт
  до файла не дотянется. В этом случае переходим на **v2 контракта**: отдаёте
  либо `mask_png_base64` прямо в JSON, либо отдельный эндпоинт
  `GET /mask/<id>`. Это решение фиксируется записью в
  `orevision-app/docs/coordination/HANDOFF.md` перед интеграцией.

Для хакатона проще всего: общая локальная папка, абсолютные пути.

## 6. Как сайт проверяет ваш ответ (валидатор контракта)

На стороне сайта есть модуль `src/contract.py`. Он прогоняет **каждый** ответ
(и mock, и ваш реальный) через проверки и при нарушении выдаёт понятную ошибку,
например: `Контракт нарушен: отсутствует поле 'confidence_map'` или
`Размер маски (900x700) не совпадает с image_size (1000x800)`.

Что именно проверяется:
1. Присутствуют все обязательные поля.
2. Типы корректны (`inference_time_ms` — число, `objects` — список и т.д.).
3. `class_legend` содержит коды 0..4.
4. Файл `mask` существует и читается как изображение.
5. Значения пикселей маски входят в набор {0,1,2,3,4}.
6. Размер маски совпадает с `image_size`.
7. У каждого объекта есть `id/class/bbox/area_px/confidence`, `class` ∈ 1..4,
   `bbox` из 4 чисел, `confidence` ∈ [0,1].

Если хотите проверить свой ответ локально — можно вызвать валидатор напрямую:
```python
from src.contract import validate_ml_response
errors = validate_ml_response(your_json_dict)
print(errors)   # пустой список = всё по контракту
```

## 7. Мини-чеклист «готовности» ML-сервиса

- [ ] `POST /analyze` принимает `multipart/form-data` с полем `image`.
- [ ] Возвращает JSON со всеми полями из §4.
- [ ] Маска — 8-битный PNG со значениями пикселей 0..4.
- [ ] (v2) `patch_grid.labels` — PNG rows×cols, коды 0..4; `mask` = его
      nearest-апскейл до `image_size`.
- [ ] `image_size` совпадает с размером маски.
- [ ] `objects` — список (можно пустой), у каждого корректные поля.
- [ ] `confidence` и `confidence_map` заполнены осмысленно.
- [ ] Ответ проходит `validate_ml_response(...)` без ошибок.
- [ ] (Желательно) есть `GET /health`.

Как только всё отмечено — сайт переключается на вас одной настройкой
`OREVISION_ML_MODE=real`, и ничего в коде сайта менять не нужно.

## 8. Версия контракта

Текущая версия: **v2** (patch-classification: блочная `mask` + `patch_grid`).
Обратная совместимость с v1 сохранена — пиксельный ML без `patch_grid` остаётся
валидным (только предупреждение). Любое изменение согласуется через
`orevision-app/docs/coordination/HANDOFF.md` и правку `API_CONTRACT.md`.
