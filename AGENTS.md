# AGENTS.md — как работают два агента в одном репозитории

Этот файл читают **оба ИИ-агента** (агент Анастасии и агент сокомандника)
в начале каждой рабочей сессии. Он описывает, кто за что отвечает, как не
мешать друг другу и как обмениваться информацией через сам репозиторий.

> Коротко: репозиторий — это наша **общая память и почта**. Мы не в одном
> чате, поэтому всё важное пишем в файлы `docs/coordination/*`, а код —
> в свои ветки. Git синхронизирует и то, и другое.

---

## 1. Два потока работы (разделение труда)

Мы делим `orevision-app` на два почти независимых потока. Граница выбрана так,
чтобы агенты редко трогали одни и те же файлы.

### Поток A — «Core & Analysis» (ядро и логика)
Отвечает за то, что происходит с данными.
- `src/schemas.py` — общие структуры (SEAM, см. §3)
- `src/config.py` — пороги, цвета, пути
- `src/ml_client.py` — HTTP-клиент к ML (mock/real)
- `mock_ml/generator.py` — имитация ответов ML
- `src/metrics.py` — площади → проценты
- `src/classification.py` — rule-based геологика
- `src/pipeline.py` — склейка ML→метрики→классификация
- `src/reports.py`, `src/storage.py` — экспорт и локальное хранение
- `tests/` — тесты логики
- `API_CONTRACT.md` — договор с ML-командой

### Поток B — «UI & Viewer» (интерфейс и вьюер)
Отвечает за то, что видит геолог.
- `app.py` — главный экран Streamlit
- `ui/viewer.py` — overlay масок, слои, (далее) zoom/pan/minimap
- `ui/components.py` — карточки, таблицы, легенда
- Многостраничность (`pages/`), batch-экран в UI, экспертный редактор маски
- Работа с большими панорамами: tiled-загрузка, навигатор, прогресс

**Правило хорошего тона:** если нужно изменить файл из чужого потока — не
делай молча. Оставь запись в `HANDOFF.md` (см. §4) или сделай так, чтобы
изменение было аддитивным и не ломало чужой код.

## 2. Ветки Git (обмен кодом)

Один общий **private**-репозиторий на GitHub. У каждого агента своя ветка.

```
main                      # стабильное, всегда запускается. Только через PR.
├── stream-a/<задача>     # ветки агента A (напр. stream-a/reports-pdf)
└── stream-b/<задача>     # ветки агента B (напр. stream-b/viewer-zoom)
```

Цикл работы агента:
1. `git checkout main && git pull` — взять свежак.
2. `git checkout -b stream-a/<задача>` — своя ветка под конкретную задачу.
3. Сделать изменения **только в файлах своего потока** (см. §1).
4. Прогнать проверки (§5).
5. `git commit` маленькими понятными коммитами → `git push -u origin <ветка>`.
6. Открыть Pull Request в `main`. Второй агент/человек смотрит и мержит.

**Никогда не коммить прямо в `main`.** Так мы не затираем работу друг друга.

## 3. SEAM — единственная общая точка кода: `src/schemas.py`

Поток A производит объекты `AnalysisResult / Metrics / Classification`,
поток B их потребляет и рисует. Пока структура этих объектов стабильна —
агенты работают параллельно, не глядя в код друг друга.

Поэтому `src/schemas.py` — **священный файл**:
- менять его можно, но каждое изменение = смена контракта между агентами;
- порядок: (1) запись в `HANDOFF.md` с описанием «что и зачем меняю»,
  (2) правка `schemas.py`, (3) уведомление второго потока в `HANDOFF.md`.

Если поток B нужно новое поле для отрисовки — он не лезет в `metrics.py`,
а просит поток A добавить поле в `Metrics` через `HANDOFF.md`.

## 4. Обмен информацией через файлы (наша «почта»)

Мы не видим чатов друг друга. Всё, что нужно передать, живёт в
`docs/coordination/`:

| Файл            | Назначение                                                        |
|-----------------|-------------------------------------------------------------------|
| `BOARD.md`      | доска задач: кто что делает, статус (todo/doing/done), владелец    |
| `HANDOFF.md`    | журнал решений и сообщений между потоками (хронологический)        |
| `PLAN_AGENT_A.md` | детальный план потока A по шагам                                 |
| `PLAN_AGENT_B.md` | детальный план потока B по шагам                                 |

Правила:
- Начинаешь задачу → отметь в `BOARD.md` (перенеси в «В работе», впиши владельца).
- Принял решение, влияющее на другого → добавь запись в `HANDOFF.md`
  (дата, автор, «что решено», «что это меняет для другого потока»).
- Закончил → в `BOARD.md` статус «Готово» + ссылка на PR.
- Эти файлы меняют **оба** агента, поэтому пиши только в конец (append),
  не переписывай чужие строки — так почти не бывает merge-конфликтов.

## 5. Обязательные проверки перед push

Любой агент перед пушем прогоняет:

```powershell
python -m py_compile app.py batch_process.py src\*.py ui\*.py mock_ml\*.py
pytest -q                 # тесты логики (поток A обязан держать их зелёными)
```

И хотя бы раз руками проверяет сквозной сценарий:
```powershell
streamlit run app.py      # открыть http://localhost:8501, нажать «демо-образец»
```

Если что-то красное — не мержим в `main`.

---

## 6. Что НЕ трогаем

- Каталог `data/` — это локальные данные, не код. В Git не попадает (`.gitignore`).
- Репозиторий `orevision-ml` — чужой. Мы общаемся с ним только по `API_CONTRACT.md`.
- Не пишем код обучения нейросетей. ML — внешний сервис за HTTP.

---

## 7. Быстрый старт агента (чек-лист на каждую сессию)

1. Прочитать этот `AGENTS.md`, свой `PLAN_AGENT_*.md`, свежий `HANDOFF.md`.
2. Посмотреть `BOARD.md` — взять следующую задачу своего потока.
3. Создать ветку, сделать задачу в границах своего потока.
4. Проверки (§5) → коммит → push → PR.
5. Обновить `BOARD.md` и при необходимости `HANDOFF.md`.
