# OreVision — локальный анализ полированных шлифов руды

Локальное веб-приложение для геологической лаборатории. Геолог загружает
OM-изображение шлифа и получает цветовую маску фаз, количественные метрики и
итоговую классификацию руды по прозрачным геологическим правилам. Всё работает
**локально**, данные не уходят в интернет.

```
Браузер ──▶ OreVision (Streamlit, localhost:8501) ──▶ ML-клиент ──▶ ML-сервис (FastAPI, localhost:8001)
                     │                                                     ▲
                     └── сам считает проценты и класс руды (rule-based) ───┘  (mask/objects/confidence)
```

> **Кто что считает.** ML отдаёт только «сырые факты о пикселях» (маска,
> объекты, площади, уверенность). Проценты и класс руды считает **сайт** —
> открытым кодом, который может прочитать геолог (`src/classification.py`).

## Принцип работы

OreVision совмещает **две независимые модели**, которые вместе решают задачи
сегментации и классификации шлифа:

1. **цветовая сегментационная модель** — попиксельно выделяет тальк (и,
   опционально, сульфидную/оксидную фазу) классическими методами
   компьютерного зрения, без обучения и без GPU;
2. **классификатор доменных областей** — на тайлах 512×512 относит область
   к сорту руды (оталькованная / рядовая / тонкая) либо к фону.

Результаты агрегируются в единую разметку изображения: количественная оценка
доли талька приходит от цветовой модели, сорт руды по областям — от
классификатора; тайлы, попавшие под маску фона, исключаются из подсчёта
сортов автоматически.

### Идентификация талька (цветовая сегментация)

Тальк в шлифах бесцветен, обладает совершенной спайностью и низким
рельефом — на снимках это тёмные слабоструктурные агрегаты с низким
контрастом и плохо выраженными границами. Модель (`talc_cosmetic/seglib.py`)
анализирует распределение цветового пространства «гладких» по цвету участков
и размечает фазы, сверяясь с одной из **референсных цветовых палитр
микроскопа** (`панорамная` / `жёлтая`, см. `talc_cosmetic/palettes/`,
автовыбор — по ошибке квантования): тальк устойчиво соответствует наиболее
тёмной фазе, тогда как высокая яркость и оттенок (желтизна/серость/
оливковость/розовость) характеризуют сульфидную и оксидную (магнетит) фазы.

> ⚠️ **Нестандартная цветовая палитра.** Автовыбор калибруется ровно на эти
> два референса. Если снимок сделан на микроскопе/камере/освещении с заметно
> другой цветопередачей (другой баланс белого, другая схема окраски фаз и
> т.п.), автоопределение может подобрать неподходящую палитру — тогда
> подсветка талька будет давать некорректные или шумные результаты. Это
> **косметический** слой (галочка «подсветка талька» на главной, включается
> ПОСЛЕ основного анализа) — он не влияет на блочную маску/метрики/класс
> руды из основного пайплайна, но при подозрении на нестандартную палитру
> его результат стоит проверять визуально, не полагаясь на него как на
> количественный.

Попиксельная маска талька проходит **плотностную агрегацию**: маска
сглаживается гауссовым ядром, по порогу плотности формируются сплошные
зоны, внутренние разрывы заливаются, мелкие изолированные компоненты
отбрасываются. Итоговая область даёт количественную оценку доли талька и
основание для правила «доля талька > 10% → оталькованная руда». Работает
скользящим окном классическими методами CV (без обучения и GPU), поэтому
справляется с гигапиксельными панорамами.

### Классификация доменных областей (сорт руды + фон)

Поверх энкодера `se_resnext50_32x4d`, предобученного на MicroNet¹ (1,2 млн
микроскопных изображений материалов, NASA) — вместо стандартных ImageNet-весов
это даёт признаки, релевантные именно для минерального анализа — обучены
**две лёгкие головы на ЗАМОРОЖЕННОМ энкодере** (одна и та же модель,
`ml_service/model.py`):

- **голова сорта** — `AdaptiveAvgPool2d` → `Linear(2048→256→3)` (ReLU,
  Dropout 0.3) → «оталькованная» / «рядовая» / «тонкая» (труднообогатимая);
- **голова фона** — `Linear(2048→1)`, бинарный детектор «фон / нерудная
  матрица» (монтировочная смола, рамка, масштабная линейка). Включена
  каскадом (`infer_batch_cascade`): если `sigmoid(bg_head) > порога` — тайл
  фон, иначе берётся argmax головы сорта. Обе головы читают ОДИН общий
  2048-эмбеддинг — энкодер прогоняется один раз на тайл.

Класс «фон» позволяет автоматически исключать нерудные области без
дополнительной пост-обработки. Обучение шло в два этапа: сперва голова при
замороженном энкодере, затем разморозка при сниженной скорости обучения;
функция потерь — взвешенная кросс-энтропия с повышенным весом редких
классов.

Метрики на отложенной выборке (235 снимков, оценка ML-команды):

| Модель | Точность | F1 «оталькованная» | F1 «рядовая» | F1 «тонкая» | F1 «фон» | Суммарный F1 |
|---|---|---|---|---|---|---|
| Классификатор доменных областей (4 класса) | 0.90 | 0.776 | 0.881 | 0.899 | 1.0 | 0.877 |

(вшитый чекпоинт `grade_unfreeze_best.pth` отдельно на 3-классовой задаче
сорта, без учёта фона, даёт macro-F1 = 0.944 — числа не противоречат друг
другу, это разные срезы оценки.)

### Тайлинг гигапиксельных панорам

Изображение разбивается на тайлы 512×512 (train-FOV модели), каждый
классифицируется независимо:

- **сетка** (`grid`, по умолчанию) — непересекающиеся тайлы, быстрее
  (≈5 Мпикс/с; для ориентира — снимок 10 000×10 000 px обрабатывается около
  20 секунд);
- **скользящее окно** (`slide`) — тайлы перекрываются на 50% (шаг 256 px),
  вероятности в зонах перекрытия усредняются перед вычислением итогового
  класса — устраняет блочные артефакты на границах тайлов ценой скорости.

### Активное обучение

Если качество модели на ваших снимках недостаточное — её можно дообучить
«на лету», не выходя из сайта: эксперт правит разметку (лассо + верный
класс, включая явную правку «это фон») → обе головы дообучаются поверх
замороженного энкодера и сохраняются **одним мультиголовым чекпоинтом**
(`ORE_ML_CKPT` подключает и сорт, и фон сразу) → снимок сразу
переанализируется дообученной моделью («было/стало»). Подробности и
инварианты — в `CLAUDE.md` и `src/active_learning.py`.

> ⚠️ **Важно.** Если результаты модели на ваших данных неудовлетворительны —
> это ожидаемо для нестандартных снимков (другой микроскоп, другая руда,
> другая цветопередача) и **устраняется дообучением** через доразметку и
> активное обучение, а не является поводом считать инструмент нерабочим.

¹ J. Stuckner, B. Harder, T. M. Smith. *Microstructure segmentation with deep
learning encoders pre-trained on a large microscopy dataset.* npj
Computational Materials, vol. 8, no. 1, Sep. 2022.
doi: 10.1038/s41524-022-00878-5.

## 1. Быстрый старт (Windows, режим разработки)

Нужен установленный **Python 3.11+** и **VS Code** (по желанию).

```powershell
# 1. Клонировать репозиторий и войти в папку
git clone <URL-репозитория> orevision-app
cd orevision-app

# 2. Создать виртуальное окружение (изолированные библиотеки проекта)
python -m venv .venv
.venv\Scripts\activate            # активация venv в PowerShell

# 3. Установить зависимости
pip install -r requirements.txt

# 4. Запустить сайт
streamlit run OreVision.py
```

Откроется браузер на `http://localhost:8501`. Загрузите изображение шлифа —
его проанализирует **встроенная модель** классификации сортов руды
(`ml_service/grade_unfreeze_best.pth` + `ml_service/bg_head_best.pth`, вшиты
в репозиторий). Индикатор модели в сайдбаре показывает имя реально
загруженного чекпоинта (вшитого или подключённого через `ORE_ML_CKPT`).

MOCK-режима больше нет: по умолчанию модель считает **в процессе сайта**
(`OREVISION_ML_MODE=local`), отдельный сервер не нужен. Нужен только torch-стек
из `requirements.txt` (тяжёлый; можно поставить CPU-сборку torch).

## 2. Вынести инференс в отдельный сервис (опционально)

По умолчанию всё считается локально. Если инференс хочется вынести на отдельную
(например, GPU) машину — поднимите сервис из этого же репозитория. Сервису
дополнительно нужен Flask, которого нет в корневом `requirements.txt`:

```powershell
pip install -r ml_service/requirements.txt   # добавляет Flask (и то, что уже не хватает)
python ml_service/server.py               # слушает :8001
$env:OREVISION_ML_MODE = "real"           # сайт ходит в сервис по HTTP
streamlit run OreVision.py
```

Контракт запроса/ответа описан в [`API_CONTRACT.md`](API_CONTRACT.md).
Он одинаков для local и real — поэтому больше ничего менять не нужно.

Не обязателен для баллов, но удобно показать «одной командой» на любой машине.

docker compose up --build

## 6a. Хранилище датасетов и разметки (active learning)

«Пакетная обработка» (очередь импорта) и «Разметка эксперта» (лассо-выделение
+ подпись участков) используют общий формат хранения на диске
(`src/dataset_storage.py`). «Разметка эксперта» размечает показанный кадр
целиком одним ROI (`roi.json` с `"kind": "whole_image"`, x=0,y=0 на весь
показанный — уменьшенный для больших панорам, как и везде в приложении —
кадр), без отдельного шага вырезания прямоугольника:

```
data/datasets/<dataset_id>/
  images/<image_id>.<ext>            — управляемая КОПИЯ исходника (picker-источники;
                                        для "путь к папке" копии нет — храним ссылку,
                                        исходный файл никогда не изменяется)
  manifest.jsonl                     — реестр изображений (формат/размер/разрешение/
                                        источник/валидность)
  annotations/<image_id>/
    image_meta.json
    regions/<region_id>/
      roi.json                — x, y, width, height ОТНОСИТЕЛЬНО исходной панорамы
                                 + coordinate_system + created_at + статус + ревизия
      roi_image.png            — вырезанный участок (копия пикселей, lossless)
      semantic_mask.png        — 8-bit ОДНОКАНАЛЬНАЯ PNG, значение пикселя = id класса;
                                  размеры точно равны roi_image.png
      annotation_state.json    — status, revision, updated_at, author, class_pixel_counts
      shapes.geojson           — подписанные лассо-контуры (класс + свободная подпись
                                  в properties каждой фигуры)
      revisions/rev_NNNN_.../   — снимок ПРЕДЫДУЩЕЙ маски+состояния перед КАЖДЫМ
                                   явным сохранением (принятая разметка не теряется)
  exports/active_learning/<export_id>/
    images/<sample_id>.png  masks/<sample_id>.png
    manifest.csv  manifest.jsonl  classes.json

data/logs/
  app_events.jsonl     — импорт, ошибки, сохранения разметки, экспорт
  batch_events.jsonl   — старт/прогресс/завершение пакетной обработки
```

Классы разметки — **не хардкод**, а `configs/annotation_classes.json`
(id, машинное имя, русское название, цвет). По умолчанию: тальк (синий),
обычные срастания (зелёный), тонкие срастания (красный), неопределённая
область (жёлто-оранжевый), неразмеченная область (прозрачный).

Статусы разметки: `draft` → `reviewed` → `accepted_for_training` /
`needs_expert_review`. Экспорт для дообучения (кнопка «📦 Экспортировать
подтверждённые разметки») берёт **только** `accepted_for_training`.

Ничего из `data/` (включая `data/datasets` и `data/logs`) в Git не попадает —
см. `.gitignore`.

### Второй вариант сохранения: «как S2_v2»

По отдельному запросу добавлен ещё один вариант сохранения результатов —
структура папок приблизительно как в примере стороннего датасета `S2_v2`
(лежит только локально в `S2_v2/`, в Git не коммитится — см. `.gitignore`):

```
imgs/<name>.jpg            — исходное изображение
masks/<name>.png           — маска классов, R=G=B=id (как masks/*.png в S2_v2)
masks_colored/<name>.png   — непрозрачная цветная маска (для быстрого визуального контроля)
masks_human/<name>.jpg     — триптих source|overlay|annotation + легенда (проверка человеком)
```

Реализовано в `src/dataset_export.py` (`export_s2_bundle`, `zip_directory`,
принимает элементы ленивым генератором — не держит все изображения батча в
памяти одновременно) и подключено как дополнительная кнопка «📦 …(формат
S2_v2)» в трёх местах:
- главная страница — экспорт текущего результата и экспертных исправлений;
- «Пакетная обработка» — экспорт всех обработанных изображений одним zip;
- «Разметка эксперта» — второй вариант экспорта active learning датасета
  (`dataset_storage.export_active_learning_s2_style`, отдельная ветка
  `exports/active_learning_s2v2/`, не пересекается с классическим форматом).

## 6. Частые проблемы

- *«streamlit не найден»* — не активировано venv (`\.venv\Scripts\activate`)
  или не установлены зависимости (`pip install -r requirements.txt`).
- *PowerShell не даёт активировать venv* — выполните один раз:
  `Set-ExecutionPolicy -Scope CurrentUser RemoteSigned`.
- *Красный индикатор модели* — в local-режиме не установлен torch
  (`pip install -r requirements.txt`) или нет файла весов
  `ml_service/grade_unfreeze_best.pth`; в real-режиме не поднят
  `python ml_service/server.py` (не забудьте `ml_service/requirements.txt` — там Flask).
- *«ModuleNotFoundError: No module named 'flask'» при запуске `ml_service/server.py`* —
  корневой `requirements.txt` не включает Flask, он только в
  `ml_service/requirements.txt` (см. раздел 2) — поставьте его отдельно.
- *Модель показывает результаты хуже ожидаемых на ваших снимках* — см.
  «Принцип работы → Активное обучение»: дообучите модель через доразметку,
  а не считайте это поводом отказаться от инструмента.
- *Странные/шумные результаты подсветки талька при визуально нормальном
  снимке* — вероятно нестандартная цветовая палитра (см. «Принцип работы →
  Идентификация талька»); это влияет только на косметический слой, не на
  основную классификацию.
- *Большая панорама (проверено до ~570 Мп)* — сайт грузит уменьшенное превью
  (≤2600px по стороне) с zoom/pan и minimap; детальный просмотр участка в
  исходном разрешении — кнопка «🔬 Инспектор участка». Расчёт метрик по маске
  для панорам >10000px идёт по тайлам, не грузя всё в память.
